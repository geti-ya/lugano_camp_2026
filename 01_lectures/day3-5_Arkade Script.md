
# 1. 발표자와 주제

- **발표자:** 마크 벤테리 (Mark Benteri / 활동명 Tiero, Ark Labs CEO 겸 공동 창립자)
- **주제:** 아케이드(Arkade) 프로토콜을 활용해 비트코인의 기본 시스템을 바꾸지 않으면서 거래 속도를 높이고 기능을 확장하는 방법

<br>

# 2. 핵심 내용 요약 3줄

1. **아케이드의 원리:** 비트코인 거래를 발생할 때마다 메인 네트워크(채굴자)에 바로 보내지 않고 밖에서 모아두었다가 한 번에 처리하여, 비싼 수수료와 느린 속도 문제를 해결했다.
2. **편리한 호환성:** 기존의 빠르고 저렴한 결제망인 '라이트닝 네트워크'를 굳이 직접 연동하거나 복잡하게 설정하지 않아도, 두 네트워크 간에 서로 자연스럽게 돈을 주고받을 수 있다.
3. **스마트 계약의 부활:** 과거 비트코인에서는 너무 느리고 비싸서 사실상 불가능했던 '에스크로(안전 결제)' 같은 복잡한 거래 조건을 아케이드를 통해 빠르고 저렴하게 실행할 수 있다.

<br>

# 3. 용어 사전

: 수업 듣는데 자주 나왔던 용어였거나 검색했던 용어들

|용어|정의|
|---|---|
|**Arcade (Ark)**|비트코인 기반의 2계층 확장성 프로토콜로, 방송되지 않은 탭루트 트랜잭션 배치를 통해 오프체인 결제를 처리함.|
|**Operator (운영자)**|Arcade 시스템에서 유동성을 제공하고 사용자 트랜잭션을 배치(Batch)로 묶어 관리하는 핵심 주체.|
|**Delegate (대리인)**|사용자가 오프라인 상태일 때 자산 갱신(Renewal) 프로세스를 대신 수행하도록 권한을 위임받은 신뢰 최소화된 제3자.|
|**Taproot (탭루트)**|비트코인의 업그레이드 기술로, 스크립트의 프라이버시를 높이고 복잡한 조건부 지출(Tapscript)을 효율적으로 지원함.|
|**vUTXO (가상 UTXO)**|Arcade 오프체인 레이어에서 존재하는 미사용 트랜잭션 출력값으로, 메인넷 블록에 기록되기 전의 가상 자산 단위.|
|**Batching (배칭)**|수천 개의 개별 트랜잭션을 하나의 비트코인 메인넷 트랜잭션으로 압축하여 기록함으로써 체인 공간을 절약하는 기술.|
|**Renewal (갱신)**|사용자가 오프체인 자산의 소유권과 보안 상태를 유지하기 위해 정기적으로 새로운 배치 상태에 동의하고 서명하는 과정.|
|**Time-lock (타임락)**|특정 시간이나 블록 높이가 지나기 전까지 트랜잭션 집행을 유예하는 조건. Arcade의 비상 탈출구(Exit) 역할을 함.|
|**Hashlock (해시락)**|특정 해시값의 원본 데이터(Pre-image)를 제시해야 자산을 지출할 수 있게 하는 조건. 라이트닝 네트워크 스왑에 주로 사용됨.|
|**Covenants (코버넌트)**|비트코인이 전송될 주소나 조건을 제한하는 스마트 계약 기능. Arcade 스크립트에서 실험적으로 구현됨.|
|**Mempool (메모리 풀)**|채굴자가 블록에 포함하기 전, 네트워크에 방송된 대기 중인 트랜잭션들이 머무르는 공간.|
|**Pre-confirmed (사전 확정)**|트랜잭션이 비트코인 블록에 포함되기 전, 운영자가 오프체인에서 결제의 유효성을 먼저 승인한 상태.|

# 4. 발표자가 특히 강조한 포인트나 인상적인 발언 1~2개

- **"아케이드는 대단하고 복잡한 다른 무언가가 아닙니다. 단지 '채굴자에게 당장 알리지 않기로 한 단순한 비트코인 거래'일 뿐입니다."**
    - 발표자는 아케이드가 비트코인의 기본 규칙이나 구조를 전혀 건드리지 않고, 비트코인 그 자체를 사용하면서도 기능을 무한히 넓힐 수 있다는 점을 거듭 강조했다.
- **"아케이드 오퍼레이터(서버)를 운영하는 것은 라이트닝 노드를 운영하는 것보다 훨씬, 훨씬 더 어렵습니다."**
    - 수많은 사람의 거래 내역을 하나로 압축하고 거대한 자금을 문제없이 관리해야 하므로, 방구석에서 컴퓨터 한 대로 쉽게 돌릴 수 있는 수준이 아니라 고도의 기술력과 인프라가 필요하다며 기술적 한계와 현실을 솔직하게 인정한 점이 인상적이다.

<br>

# 5. 흥미로웠던 영역 추가설명

적고 보니 아케이드 스크립트에 관한 이야기가 빠진 것 같아, 간단한 설명과 개인적으로 궁금해서 AI에게 물어본 내용을 첨부한다.

**아케이드 스크립트는 비트코인 메인 네트워크 내부의 스마트 컨트랙트가 아니라, 아케이드 오퍼레이터(서버) 환경에서 작동하도록 고안된 '자체적인 프로그래밍 언어이자 컴파일러'다.**

서버(오퍼레이터) 이야기가 강연에서 큰 비중을 차지할 수밖에 없는 이유와 아케이드 스크립트의 구체적인 특징은 다음과 같다.

1. 아케이드 스크립트란 무엇인가?

	- 강연자는 아케이드 스크립트를 **"추가적인 연산 코드(extra op codes)가 더해진 비트코인 스크립트"**라고 정의한다.
	- 기존 비트코인의 복잡하고 낮은 수준(low-level)의 바이트 코드를 직접 작성하는 대신, 개발자들이 더 직관적인 **고수준(high-level) 언어로 스마트 컨트랙트를 짤 수 있도록 돕는 컴파일러 환경**을 제공한다.
	- 가장 큰 목적은 개발자들이 비트코인에 향후 도입될 가능성이 있는 '커버넌트(Covenants)' 같은 미래 업그레이드 기능을 아케이드 상에서 미리 실험해볼 수 있게 하는 것이다.

2. 왜 스크립트 강연에서 '서버' 이야기가 필수적인가?

	- 아케이드 스크립트에 추가된 '새로운 연산 코드'들은 아직 실제 비트코인 채굴자들이 인식하거나 검증할 수 없는 비공식 코드다.
	- 따라서 이 코드를 읽고 실행해줄 별도의 주체가 필요한데, **오직 아케이드 서버(오퍼레이터)만이 이 코드를 인식하고 오프체인(Off-chain) 상태에서 실행(emulate)**해준다.
	- 사용자는 블록체인의 검증 대신, 오퍼레이터 서버가 이전 상태를 조작하지 않고 코드를 정직하게 실행해줄 것이라는 신뢰(Trust assumption)를 바탕으로 이 스크립트를 이용한다.
	- 강연자가 **"아케이드 서버(오퍼레이터)를 운영하는 것은 라이트닝 노드를 운영하는 것보다 훨씬 어렵다"**라고 단언한 이유도 여기에 있다. 수많은 사용자의 복잡한 오프체인 스마트 컨트랙트를 대신 처리하고, 만약의 사태에 대비해 신뢰 실행 환경(TEE)을 구축해 막대한 유동성을 완벽하게 관리해야 하는 주체가 바로 서버이기 때문이다.

	**요약**: 아케이드 스크립트는 비트코인의 미래 기능을 미리 써볼 수 있게 만든 '확장된 언어'이고, 이 언어로 작성된 계약을 비트코인 밖에서 연산·실행해주는 핵심 인프라가 '아케이드 서버'다. 그래서 언어(스크립트)의 가능성을 설명하려면 필연적으로 그것을 실행하는 서버의 역할과 어려움이 함께 강조될 수밖에 없다.

3. 왜 메인 네트워크(온체인)가 아닌 별도 환경(오퍼레이터 서버)에서 작동하게 만들었나?

	- **비싼 수수료와 느린 속도 극복 (확장성)**: 비트코인 메인 네트워크는 블록 생성에 약 10분이 걸리고 공간이 매우 제한적이다. 과거 수수료가 300~400달러까지 치솟은 적도 있어, 50달러짜리 신발을 사거나 5달러어치 비트코인을 구매하는 소액 상거래에 메인 네트워크의 스마트 컨트랙트(에스크로 등)를 쓰는 것은 비용상 전혀 맞지 않는다.
	- **오프체인 대량 압축(Batching)**: 오퍼레이터 서버를 거치면 수만 명의 거래를 모아두었다가 다음 비트코인 블록에 **단 하나의 거대한 온체인 거래로 압축**해 기록할 수 있다. 이를 통해 네트워크 수수료를 획기적으로 줄이고 거의 즉각적인 거래 경험을 제공한다.
	- **기본 레이어(Base Layer)를 건드리지 않기 위함**: 비트코인의 기본 규칙이나 기반 레이어를 전혀 수정하지 않고도 기능을 확장하기 위해 이 방식을 택했다.
	- **합의 없이 미래 기술 선행 실험**: '커버넌트' 같은 미래 기술을 메인 네트워크에 적용하려면 비트코인 생태계 전체의 합의가 필요하다. 하지만 독립된 아케이드 오퍼레이터 환경을 사용하면 남을 설득할 필요 없이 **오퍼레이터 서버만 인식하는 '추가 연산 코드'를 임의로 추가**해, 개발자들이 곧바로 새로운 스마트 컨트랙트 기능을 실험하고 구축해볼 수 있다.
